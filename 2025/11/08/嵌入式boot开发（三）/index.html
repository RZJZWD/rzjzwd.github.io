嵌入式boot开发（三）

发表于 2025-11-08 更新于 2025-11-19 分类于 + Bootloader
嵌入式bootloader开发：boot通信协议设计及boot程序微调

前言

在上一篇文章嵌入式boot开发（二）中我们实现了boot状态机并对一些函数进行优化，这个章节就来设计一个上位机与boot通信的通信协议。

想一下

如果在我们设计最简单的boot时，一个boot程序主要有两个功能：

  • 可由外部信号选择来进入bootloader模式
  • 跳转app的功能
    如果想要boot具有更新固件的能力，则需要boot能够读取并写入固件，所以还要一个功能就是上传固件

通信数据约定

为了可靠传输，我选择使用半双工通信，下位机(MCU一下统称下位机)被动的接收上位机指令，同时使用自定义命令帧来增强传输可靠性

命令帧

命令帧结构为: [0xAA] [0x55] [命令字(1byte)] [数据长度(小端序 2byte)] [数据] [帧校验数据(2byte)]

  • 0xAA和0x55为帧头

  • 命令字

    命令名 功能 命令字
    EnterBoot 进入boot模式 0x01
    Upload 上传固件 0x02
    Verify 验证固件 0x03
    RunApp 运行app 0x04
    Ack 肯定应答 0x05
    Nack 否定应答 0x06
    ErrorResponse 错误响应 0x07

  • 数据长度采用小端序，方便下位机解析

  • 数据，任意长度

  • 帧校验数据，从命令字到数据结束的累加和取反

固件包

固件通常都是十几或者几十KB，所以这里采取分包传输的策略，将一个固件按照预先设定的包大小进行拆分；比如一个19KB的固件，按照1KB的包大小拆分成19个包。
在上位机向下位机传数据包时，为了防止数据包在传输过程中因为种种可能的干扰导致报数据出错，这里采用CRC32校验来用于下位机确认包完整性
因此一个包数据的构成为: [当前包号] [总包号] [包数据(假设为1KB)] [包数据的CRC32]

  • 注：为了与CRC32长度相同，处理包数据其他都是4byte的大端数据

上位机发送

为了配套使用，我写了一个简单的上位机并开源在github，由于是边学c#边写的，其中还充斥了大量ai生成代码，比较丑陋，望不吝赐教
此上位机目前实现了：

  • 基础串口通信:
    • 自动发送
    • 保存接收为txt
    • 收发方标记
  • boot模式:
    • 发送命令，含传输层超时重发和调度层追加重发
    • 读取固件并自动分包计算CRC32
    • 测试按钮，用于测试

下位机接收

上位机只是添头，主要是下位机的接收该怎么写。这里还是使用状态机的方法，

状态制定

思考一下命令帧，一共有五个部分组成: 帧头，命令字，数据长度，数据和校验。也就是说当我们接收字节数据时就是围绕着这几个状态

但是这里我把它再细分了一下，将帧头拆为两次，数据长度也拆开，这主要是为了配合按字节接收

状态转换

让我们一起分析一下接收流程：

  1. RX_STATE_HEADER1 ：假设先收到0xAA，状态由 RX_STATE_HEADER1 -> RX_STATE_HEADER2 ；否侧状态回到 RX_STATE_HEADER1 继续等待帧头

  2. RX_STATE_HEADER2 ：收到0x55，说明帧头接收完了，状态由 RX_STATE_HEADER2 -> RX_STATE_CMD ；否侧状态回到 RX_STATE_HEADER1 继续等待帧头

  3. RX_STATE_CMD ： 检测接收的字节数据是否在命令字枚举范围内，命令字合法则进入下一状态 RX_STATE_LEN_LOW ；否侧状态回到 RX_STATE_HEADER1 继续等待帧头

  4. RX_STATE_LEN_LOW ：接收长度数据的低位，然后进入 RX_STATE_LEN_HIGH

  5. RX_STATE_LEN_HIGH ：接收长度数据的高位，这时候会有分支，如果长度为零则直接进入 RX_STATE_CHECKSUM ，不为零且小于数据缓冲区长度则进入 RX_STATE_DATA ; 否侧状态回到 RX_STATE_HEADER1 继续等待帧头

  6. RX_STATE_DATA ：接收数据，内置计数器，当接收数据不等于长度时说明接收数据未满，循环 RX_STATE_DATA ，直到接收数据等于长度转到 RX_STATE_CHECKSUM

  7. RX_STATE_CHECKSUM ：校验数据，除了帧头不加，其他累计和取反并于接受的校验数据比较，相对说明本次命令帧接收成功；否侧状态回到 RX_STATE_HEADER1 继续等待帧头

封装并分层

到这里，boot显然不适合再在main中编写了，我们需要将其独立出来作为一个模块存在。暂时将其分为boot层和boot_cmd层

  • boot_cmd层：只负责解析命令帧和构建命令帧
  • boot层：负责封装并暴露 boot_cmd 中的接口，以及一些与boot强相关的函数，例如boot状态机，app跳转等
